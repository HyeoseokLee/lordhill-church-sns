# 핵심 룰

## 단위개발 워크플로우

프롬프트에 '워크플로우'라는 용어가 포함되면 반드시 아래 4단계를 순서대로 실행한다.
일반적인 코드 작성/수정 요청에는 적용하지 않는다.

### 1. Research

- 관련된 것으로 보이는 rule, 코드베이스를 깊이 상세히 파악한다.
- 해당 구현 방법의 global standard를 충분히 탐구한다.
- 결과를 `.claude/docs/{주제}-research.md` 문서로 작성한다.
- 리서치 문서를 개발자에게 먼저 제시한다.
- **개발자가 허락한 후에만** 다음 단계로 진행한다.

### 2. Plan

- 리서치 결과를 바탕으로 기존 관례를 절대로 무시하지 않는 계획을 세운다.
- 소스 파일을 읽고 실제 코드베이스를 기반으로 플랜을 세운다.
- `.claude/docs/{주제}-plan.md`를 작성하여 보여준다.
- 플랜 구성 항목:
  - **접근방식 설명**: 왜 이 방법을 선택했는지
  - **핵심 데이터**: 관련 모델, API, 상태 등
  - **코드 스니펫**: 필요시 핵심 구현 미리보기
  - **파일 경로**: 생성/수정할 파일 목록
  - **고려사항**: 엣지케이스, 호환성, 성능 등
- **개발자에게 구현 여부를 묻고 허락한 후에만** 구현에 진입한다.

### 3. 구현

- 기존 관례를 절대로 무시하지 마라.
- 코드베이스에서 이미 사용 중인 패턴, 네이밍, 폴더 구조를 따른다.
- Research와 Plan 문서에 정리한 내용을 충실히 반영한다.

### 4. 검증

- 서브에이전트를 띄운다.
- 서브에이전트가 Research, Plan, 그리고 코드베이스를 바탕으로 구현한 코드를 평가한다.
- 평가 결과를 개발자에게 보고한다.
